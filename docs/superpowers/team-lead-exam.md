# Team Lead 上岗考（题目卷 · v2 加难版）

> 用途：验证"一个**全新、失忆的** Claude Code 主 session（Team Lead）**只靠本仓库文档**能否胜任工厂编排"。
> 凡冷启动 Lead 答错/答不出的题 = **文档缺口**，据此迭代文档、回归。答案见 `team-lead-exam-KEY.md`（考生**不得阅读**）。
> 这版不只考"是什么/在哪"，更考**深层设计决策、每个角色提示词的具体契约、编排/持久化边界、逃生通道、失败模式处置**。
> 评分：每题 0/1/2（2=正确且引对出处/能落地，1=部分对，0=错/缺）。满分 60（30 题）。
> **及格：≥50 且所有 ★PASS-gate 题（Q3 Q5 Q7 Q13 Q14 Q16 Q21 Q24 Q27）均 ≥1。**

## Tier 1 — 运营：你能把工厂跑起来吗
- **Q1**：项目现在做什么书？相比之前发生了什么转向、为什么？
- **Q2**：当前构建到哪一步？磁盘实况（state.json/artifacts/trace）与"完成度"是否一致？下一步紧要动作？
- **Q3 ★**：用户说"写第 4 章"。给出**确切**调用方式与全部参数（可直接照抄执行）。
- **Q4**：一章的 `focus / paths / highlight` 三个参数分别从哪个文件、哪个字段取？
- **Q5 ★**：怎样**不重做**已完成阶段地续跑一条流水线？发车与续跑用的 Workflow 入参（`name` vs `scriptPath`）有何不同？
- **Q6**：怎么实时看进度、必要时急停一条正在跑的流水线？
- **Q7 ★**：有哪些质量闸门？各自是**独立脚本**还是并入某脚本？分别怎么跑？（注意 linter 的真实数量）

## Tier 2 — 角色提示词的具体契约（逐角色）
- **Q8**：6 个角色分别是谁？各自 `.md` 在哪？workflow 用什么机制调用它们？为什么每个 workflow agent 提示词第一行要"先读自己的 .md"？
- **Q9**：analyst 产出的 `dossier.json` 必含哪些字段？为什么 `embed_excerpts` 必须是**逐字真实源码**而非摘要？
- **Q10 ★? **：implementer.md 里**具体哪条**契约防止"过度删减/误删"？analyst 的 `subtraction_plan` 用 `delete` 与 `must_keep` 各起什么作用、形态要求是什么？
- **Q11**：writer.md 的**五条强制契约**是什么？writer 发现精简版缺了要讲清的细节，提示词要求它怎么做？
- **Q12**：reviewer.md 的结构化反馈 schema 是什么？为什么每条 issue **必须**带 `suggested_fix`？哪些维度是 auto-REJECT？
- **Q13 ★**：tester 验证的是"精简版自洽"还是"复现真实 vLLM 行为"？为什么？它的判定是二元还是打分？

## Tier 3 — 编排与持久化边界、逃生通道（你反复追问的硬骨头）
- **Q14 ★**：本体系的"持久化"分哪两层？**什么持久、什么不持久**？为什么这样切？
- **Q15**：workflow（无状态、agent 一次性）**什么时机切换到持久/命名 agent**？具体触发条件 + 机制（谁来编排活体双向对话）？
- **Q16 ★**：系统在哪里埋了**逃生通道**？一个 workflow agent 跑到一半发现路线错，能否中止并把问题交给 Lead？具体机制是什么（它不能自己做什么、能做什么）？
- **Q17**：逃生舱共有哪几个触发 stage？各返回什么字段（注意 `reason` vs `problems` 的区别）？收到后你按 stage 分别怎么处理？
- **Q18**：dossier 阶段除了产档案还有什么**对抗性**步骤？它在何时、拦住什么？
- **Q19**：多次重做时，如何保证 fresh/重启的 agent **不失忆、不重蹈覆辙**？（列出具体机制）

## Tier 4 — 设计原理：你懂"为什么"吗
- **Q20**：旧体系（ch04 实测）是怎么"脱节"的？为什么"档案即唯一真相源"能从结构上根除它？
- **Q21 ★**：为什么跨章一致性/伏笔/回收**不能赌**一个"全程常驻 writer"的对话记忆？正确做法是什么？
- **Q22**：为什么 implementer 必须"只做减法"、且**不能自行决定**删什么？这对 writer 的工作有什么直接好处？
- **Q23**：伏笔是"碰运气涌现"还是"设计出来的"？本体系靠什么在写作前就把跨章弧线定好？

## Tier 5 — 失败模式与场景处置（你会处理硬情况吗）
- **Q24 ★**：implementer 把 writer 要讲的关键符号删了。系统**有哪几道防线**会发现并纠正？（尽量列全）
- **Q25**：writer 写出的章节大段在讲"精简版"而非真实源码。哪个 linter / 哪个评审维度会拦？依据是什么？
- **Q26**：正文里出现了 `instances/vllm/source/...` 路径、或 "Cell 3" 这种标题。会发生什么？哪个闸门拦？为什么这是问题？
- **Q27 ★**：ch20 要回收 ch04 埋下的伏笔，但写 ch20 的 writer 是全新进程、没见过 ch04。它怎么知道要回收什么？
- **Q28**：你想发 ch04，但发现 `cartography/map.json` 里这一章的 paths 不全/不确定。你怎么补齐参数再发车？
- **Q29**：某章在 reviewer↔writer 之间反复 >3 轮还过不了。系统/你怎么处理（自动 + 人工两条路）？
- **Q30**：vLLM 相关代码必须在哪运行、为什么？容器 vllm 版本与源码 pin 不一致，引用行号以谁为准、容器用来干嘛？
