1. [ENTRY] User calls `LLM.generate(prompts, sampling_params)` (offline) or HTTP POST hits the OpenAI server route вАФ `vllm/entrypoints/llm.py:LLM.generate()` or `vllm/entrypoints/openai/api_server.py` вЖТ `OpenAIServingChat.create_chat_completion()` (chat_completion/serving.py).
2. [RENDER] The prompt is rendered/normalized into an EngineInput вАФ `BaseRenderer.render_cmpl()/render_chat()` (vllm/renderers/base.py) tokenizes (AsyncMicrobatchTokenizer) and runs the multimodal processor. Offline path goes through `LLM._run_completion()`; server path calls `engine_client.generate(...)`.
3. [ASYNC FACADE] `AsyncLLM.generate(prompt, sampling_params, request_id)` (vllm/v1/engine/async_llm.py:524) is invoked; it returns an AsyncGenerator and immediately calls `AsyncLLM.add_request()` (async_llm.py:280).
4. [STAGE 1 вАФ INPUT] `add_request()` calls `InputProcessor.process_inputs()` (vllm/v1/engine/input_processor.py:234) вЖТ validates SamplingParams, tokenizes, extracts/sorts multimodal MultiModalFeatureSpec, assigns request_id (external_req_id + uuid), and builds an immutable `EngineCoreRequest` (vllm/v1/engine/__init__.py).
5. [FAN-OUT (n>1)] If `params.n > 1`, a `ParentRequest` (vllm/v1/engine/parallel_sampling.py) fans out n child EngineCoreRequests (child id `{idx}_{parent}`, unique seeds); each child goes through `_add_request()` independently.
6. [STAGE 1 вЖТ DISPATCH] `AsyncLLM._add_request()` (async_llm.py:400) does two things in parallel: (a) registers per-request state + a `RequestOutputCollector` queue in the in-process `OutputProcessor`, and (b) ships the EngineCoreRequest across the process boundary via `EngineCoreClient.add_request_async()` (vllm/v1/engine/core_client.py:1058, AsyncMPClient вЖТ ZMQ DEALER + msgpack).
7. [IPC RECEIVE] In the separate EngineCore process, `EngineCoreProc.process_input_sockets()` (vllm/v1/engine/core.py:1320) receives ZMQ frames, decodes via MsgpackDecoder (+ TensorIpcReceiver for zero-copy multimodal tensors), and enqueues onto the input_queue.
8. [STAGE 2 вАФ ENQUEUE] `EngineCoreProc.run_busy_loop()` (core.py:1164) вЖТ `_process_input_queue()` вЖТ `EngineCore.add_request()` (core.py:315) вЖТ `Scheduler.add_request()` (vllm/v1/core/sched/scheduler.py:1728) places the Request on the waiting queue.
9. [STAGE 2 вАФ SCHEDULE] The busy loop calls `EngineCore.step()` (core.py:402) вЖТ `Scheduler.schedule()` (scheduler.py:352): token-budget continuous batching, RUNNING-first with preemption, prefix-cache lookup via `kv_cache_manager.get_computed_blocks()` and block allocation via `kv_cache_manager.allocate_slots()` (vllm/v1/core/kv_cache_manager.py), producing a `SchedulerOutput`.
10. [STAGE 2 вАФ EXECUTE] `EngineCore.step()` calls `model_executor.execute_model(scheduler_output, non_block=True)` (vllm/v1/executor/abstract.py:210) вЖТ Executor (uni/mp/ray) broadcasts the RPC вЖТ `Worker.execute_model()` (vllm/v1/worker/gpu_worker.py) вЖТ `GPUModelRunner.execute_model()` (vllm/v1/worker/gpu_model_runner.py:3825).
11. [STAGE 2 вАФ INPUT TENSORS] Inside the runner: `_update_states()` (gpu_model_runner.py:1065) reconciles the persistent InputBatch against SchedulerOutput, then `_prepare_inputs()` (gpu_model_runner.py:1787) builds token/position tensors, commits the block table, and computes slot_mapping (Triton). `CommonAttentionMetadata` вЖТ per-backend metadata via `AttentionMetadataBuilder.build()`.
12. [STAGE 2 вАФ FORWARD] The transformer forward runs (vllm/model_executor/models/llama.py etc.); each `Attention` layer dispatches to its backend kernel (FlashAttention/FlashInfer/Triton, vllm/v1/attention/backends/*), reading/writing paged KV blocks. Forward result + hidden states are cached in `ExecuteModelState`; execute_model may return None (async two-phase).
13. [STAGE 2 вАФ SAMPLE] `EngineCore.step()` then calls `model_executor.sample_tokens(grammar_output)` вЖТ `GPUModelRunner.sample_tokens()` (gpu_model_runner.py:4178) вЖТ `Sampler.forward()` (vllm/v1/sample/sampler.py) (penalties вЖТ logits processors вЖТ temperature вЖТ top-k/top-p вЖТ greedy/random) or `RejectionSampler` for speculative decoding; `_bookkeeping_sync()` (gpu_model_runner.py:3397) writes sampled tokens back into the persistent batch.
14. [STAGE 2 вАФ UPDATE] `Scheduler.update_from_output()` (scheduler.py:1290) appends new tokens to each Request, runs stop/EOS detection, caches finalized KV blocks, frees finished requests, and emits a per-client dict of `EngineCoreOutputs`.
15. [IPC SEND] `EngineCoreProc.process_output_sockets()` (core.py:1466) msgpack-encodes EngineCoreOutputs and sends them over the ZMQ output socket back to the frontend process.
16. [STAGE 3 вАФ RECEIVE] `AsyncMPClient.process_outputs_socket()` (core_client.py:942) decodes frames and puts EngineCoreOutputs onto an asyncio outputs_queue; `AsyncLLM._run_output_handler()` background task (async_llm.py:637) awaits `get_output_async()`.
17. [STAGE 3 вАФ OUTPUT] The output_handler calls `OutputProcessor.process_outputs()` (vllm/v1/engine/output_processor.py:572, single batch loop): incremental detokenization (`IncrementalDetokenizer.update()`, detokenizer.py) with stop-string buffering, logprobs assembly (`LogprobsProcessor`, logprobs.py), and RequestOutput construction (DELTA vs cumulative, parent aggregation for n>1).
18. [STAGE 3 вАФ ENQUEUE] Each assembled `RequestOutput` is pushed to its per-request `RequestOutputCollector.put()` (output_processor.py:45), with delta-merging if the producer outpaces the consumer.
19. [STREAM OUT] Back in `AsyncLLM.generate()` (async_llm.py:576), the consumer drains the queue via `q.get_nowait() or await q.get()` and yields each RequestOutput; on finish or client disconnect it aborts. The server formats each as an SSE ChatCompletionStreamResponse (or a final JSON), completing the round-trip to the user.

## еЫЫгАБзЪЗеЖ†зЇІдЄУйҐШ (flagship)

- The async 3-stage decoupling: in-process input preprocessing + out-of-process EngineCore busy loop + in-process background output handler, bridged by per-request RequestOutputCollector queues (async_llm.py, the architectural heart of v1).
- Continuous batching with token budgets and dynamic preemption: phase-free token-centric scheduling, RUNNING-first allocation, and the NewRequestData/CachedRequestData persistent-batch protocol (scheduler.py + output.py).
- Paged KV cache with prefix caching: block pool, block hashing, the doubly-linked free queue, three-stage allocation, and fixed-point multi-attention cache-hit search (kv_cache_manager.py + block_pool.py + kv_cache_coordinator.py).
- The persistent InputBatch and decoupled forward/sampling via ExecuteModelState: zero-realloc continuous batching on the GPU plus two-phase execute_model/sample_tokens enabling async scheduling and PP overlap (gpu_model_runner.py + gpu_input_batch.py).
- The ZMQ+msgpack IPC boundary with zero-copy tensor transport: EngineCoreClient/EngineCoreProc, byte-tagged request protocol, and TensorIpcSender/Receiver (core_client.py + core.py + tensor_ipc.py).

### async-engine

_AsyncLLM is a 3-stage async facade that decouples input preprocessing, EngineCore scheduling/execution, and output postprocessing into concurrent independent loops, enabling non-blocking streaming inference via per-request async generators and event-driven ZMQ IPC._
