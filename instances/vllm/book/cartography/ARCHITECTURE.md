## 二、请求全生命周期主线 (the spine)

一个推理请求从入口到流式输出的完整路径，**全书的叙事主线**：

1. [ENTRY] User calls `LLM.generate(prompts, sampling_params)` (offline) or HTTP POST hits the OpenAI server route — `vllm/entrypoints/llm.py:LLM.generate()` or `vllm/entrypoints/openai/api_server.py` → `OpenAIServingChat.create_chat_completion()` (chat_completion/serving.py).
2. [RENDER] The prompt is rendered/normalized into an EngineInput — `BaseRenderer.render_cmpl()/render_chat()` (vllm/renderers/base.py) tokenizes (AsyncMicrobatchTokenizer) and runs the multimodal processor. Offline path goes through `LLM._run_completion()`; server path calls `engine_client.generate(...)`.
3. [ASYNC FACADE] `AsyncLLM.generate(prompt, sampling_params, request_id)` (vllm/v1/engine/async_llm.py:524) is invoked; it returns an AsyncGenerator and immediately calls `AsyncLLM.add_request()` (async_llm.py:280).
4. [STAGE 1 — INPUT] `add_request()` calls `InputProcessor.process_inputs()` (vllm/v1/engine/input_processor.py:234) → validates SamplingParams, tokenizes, extracts/sorts multimodal MultiModalFeatureSpec, assigns request_id (external_req_id + uuid), and builds an immutable `EngineCoreRequest` (vllm/v1/engine/__init__.py).
5. [FAN-OUT (n>1)] If `params.n > 1`, a `ParentRequest` (vllm/v1/engine/parallel_sampling.py) fans out n child EngineCoreRequests (child id `{idx}_{parent}`, unique seeds); each child goes through `_add_request()` independently.
6. [STAGE 1 → DISPATCH] `AsyncLLM._add_request()` (async_llm.py:400) does two things in parallel: (a) registers per-request state + a `RequestOutputCollector` queue in the in-process `OutputProcessor`, and (b) ships the EngineCoreRequest across the process boundary via `EngineCoreClient.add_request_async()` (vllm/v1/engine/core_client.py:1058, AsyncMPClient → ZMQ DEALER + msgpack).
7. [IPC RECEIVE] In the separate EngineCore process, `EngineCoreProc.process_input_sockets()` (vllm/v1/engine/core.py:1320) receives ZMQ frames, decodes via MsgpackDecoder (+ TensorIpcReceiver for zero-copy multimodal tensors), and enqueues onto the input_queue.
8. [STAGE 2 — ENQUEUE] `EngineCoreProc.run_busy_loop()` (core.py:1164) → `_process_input_queue()` → `EngineCore.add_request()` (core.py:315) → `Scheduler.add_request()` (vllm/v1/core/sched/scheduler.py:1728) places the Request on the waiting queue.
9. [STAGE 2 — SCHEDULE] The busy loop calls `EngineCore.step()` (core.py:402) → `Scheduler.schedule()` (scheduler.py:352): token-budget continuous batching, RUNNING-first with preemption, prefix-cache lookup via `kv_cache_manager.get_computed_blocks()` and block allocation via `kv_cache_manager.allocate_slots()` (vllm/v1/core/kv_cache_manager.py), producing a `SchedulerOutput`.
10. [STAGE 2 — EXECUTE] `EngineCore.step()` calls `model_executor.execute_model(scheduler_output, non_block=True)` (vllm/v1/executor/abstract.py:210) → Executor (uni/mp/ray) broadcasts the RPC → `Worker.execute_model()` (vllm/v1/worker/gpu_worker.py) → `GPUModelRunner.execute_model()` (vllm/v1/worker/gpu_model_runner.py:3825).
11. [STAGE 2 — INPUT TENSORS] Inside the runner: `_update_states()` (gpu_model_runner.py:1065) reconciles the persistent InputBatch against SchedulerOutput, then `_prepare_inputs()` (gpu_model_runner.py:1787) builds token/position tensors, commits the block table, and computes slot_mapping (Triton). `CommonAttentionMetadata` → per-backend metadata via `AttentionMetadataBuilder.build()`.
12. [STAGE 2 — FORWARD] The transformer forward runs (vllm/model_executor/models/llama.py etc.); each `Attention` layer dispatches to its backend kernel (FlashAttention/FlashInfer/Triton, vllm/v1/attention/backends/*), reading/writing paged KV blocks. Forward result + hidden states are cached in `ExecuteModelState`; execute_model may return None (async two-phase).
13. [STAGE 2 — SAMPLE] `EngineCore.step()` then calls `model_executor.sample_tokens(grammar_output)` → `GPUModelRunner.sample_tokens()` (gpu_model_runner.py:4178) → `Sampler.forward()` (vllm/v1/sample/sampler.py) (penalties → logits processors → temperature → top-k/top-p → greedy/random) or `RejectionSampler` for speculative decoding; `_bookkeeping_sync()` (gpu_model_runner.py:3397) writes sampled tokens back into the persistent batch.
14. [STAGE 2 — UPDATE] `Scheduler.update_from_output()` (scheduler.py:1290) appends new tokens to each Request, runs stop/EOS detection, caches finalized KV blocks, frees finished requests, and emits a per-client dict of `EngineCoreOutputs`.
15. [IPC SEND] `EngineCoreProc.process_output_sockets()` (core.py:1466) msgpack-encodes EngineCoreOutputs and sends them over the ZMQ output socket back to the frontend process.
16. [STAGE 3 — RECEIVE] `AsyncMPClient.process_outputs_socket()` (core_client.py:942) decodes frames and puts EngineCoreOutputs onto an asyncio outputs_queue; `AsyncLLM._run_output_handler()` background task (async_llm.py:637) awaits `get_output_async()`.
17. [STAGE 3 — OUTPUT] The output_handler calls `OutputProcessor.process_outputs()` (vllm/v1/engine/output_processor.py:572, single batch loop): incremental detokenization (`IncrementalDetokenizer.update()`, detokenizer.py) with stop-string buffering, logprobs assembly (`LogprobsProcessor`, logprobs.py), and RequestOutput construction (DELTA vs cumulative, parent aggregation for n>1).
18. [STAGE 3 — ENQUEUE] Each assembled `RequestOutput` is pushed to its per-request `RequestOutputCollector.put()` (output_processor.py:45), with delta-merging if the producer outpaces the consumer.
19. [STREAM OUT] Back in `AsyncLLM.generate()` (async_llm.py:576), the consumer drains the queue via `q.get_nowait() or await q.get()` and yields each RequestOutput; on finish or client disconnect it aborts. The server formats each as an SSE ChatCompletionStreamResponse (or a final JSON), completing the round-trip to the user.

## 三、跨子系统依赖图

- `entrypoints` → `async-engine` — API server / LLM facade calls AsyncLLM.generate() and add_request(); offline LLM wraps LLMEngine over EngineCore
- `entrypoints` → `input-processor` — Renderer (render_cmpl/render_chat) produces EngineInput consumed by InputProcessor
- `async-engine` → `input-processor` — AsyncLLM.add_request() calls InputProcessor.process_inputs() to build EngineCoreRequest (async_llm.py:280)
- `async-engine` → `output-processor` — output_handler task calls OutputProcessor.process_outputs() and registers per-request RequestOutputCollector queues (async_llm.py:637)
- `async-engine` → `engine-core` — EngineCoreClient (AsyncMPClient) ships EngineCoreRequest over ZMQ to EngineCore process and receives EngineCoreOutputs (core_client.py:1058,942)
- `input-processor` → `sampling` — validates and clones SamplingParams; parallel_sampling assigns per-child seeds
- `input-processor` → `model-definitions` — queries ModelConfig for max_model_len / vocab_size and task routing
- `engine-core` → `scheduler` — EngineCore.step() calls Scheduler.schedule() and Scheduler.update_from_output() (core.py:402)
- `engine-core` → `worker-and-executor` — EngineCore instantiates Executor and calls execute_model()/sample_tokens()/collective_rpc() (core.py:118,402)
- `engine-core` → `config-and-wiring` — EngineCore.__init__ receives VllmConfig; selects Scheduler class and Executor backend from config
- `scheduler` → `kv-cache` — Scheduler calls kv_cache_manager.get_computed_blocks/allocate_slots/free for block management and prefix caching (scheduler.py:465,617,961)
- `scheduler` → `spec-decode` — schedules draft tokens (scheduled_spec_decode_tokens) and validates speculative tokens via grammar/rejection during update_from_output
- `scheduler` → `output.py (SchedulerOutput)` — produces SchedulerOutput (NewRequestData/CachedRequestData/num_scheduled_tokens) consumed by model-runner
- `worker-and-executor` → `model-runner` — Worker.execute_model()/sample_tokens() delegate to GPUModelRunner (gpu_worker.py → gpu_model_runner.py:3825,4178)
- `worker-and-executor` → `distributed-parallelism` — Executor sets up parallel process groups; multiproc/ray executors drive collective RPC and PP/TP comms
- `model-runner` → `scheduler` — consumes SchedulerOutput to update persistent InputBatch (_update_states)
- `model-runner` → `kv-cache` — builds block tables / slot_mapping from scheduler-allocated block ids for paged attention
- `model-runner` → `attention` — constructs CommonAttentionMetadata → AttentionMetadataBuilder.build() per KV group before forward
- `model-runner` → `model-definitions` — runs the transformer forward (LlamaForCausalLM.forward) to produce logits/hidden states
- `model-runner` → `sampling` — sample_tokens() calls Sampler.forward()/RejectionSampler.forward() (gpu_model_runner.py:4178)
- `model-runner` → `spec-decode` — invokes proposers (ngram/EAGLE/DFlash) and rejection sampler; manages draft buffers
- `model-definitions` → `attention` — each layer instantiates v1 Attention bound to a selected AttentionBackend (llama.py → attention.py)
- `model-definitions` → `distributed-parallelism` — Column/Row/QKVParallelLinear call all-reduce/all-gather collectives during forward
- `attention` → `kv-cache` — attention kernels read/write paged KV cache blocks; backends declare kv_cache_shape/stride layout
- `spec-decode` → `sampling` — RejectionSampler wraps Sampler to verify draft tokens against target logits
- `output-processor` → `engine-core` — consumes EngineCoreOutput (new_token_ids/logprobs/finish_reason) emitted by EngineCore.step()
- `config-and-wiring` → `model-definitions` — VllmConfig drives model loading/initialization (get_model, initialize_model)
- `config-and-wiring` → `distributed-parallelism` — ParallelConfig selects executor backend and parallelism sharding (TP/PP/DP/EP)

## 四、皇冠级专题 (flagship)

- The async 3-stage decoupling: in-process input preprocessing + out-of-process EngineCore busy loop + in-process background output handler, bridged by per-request RequestOutputCollector queues (async_llm.py, the architectural heart of v1).
- Continuous batching with token budgets and dynamic preemption: phase-free token-centric scheduling, RUNNING-first allocation, and the NewRequestData/CachedRequestData persistent-batch protocol (scheduler.py + output.py).
- Paged KV cache with prefix caching: block pool, block hashing, the doubly-linked free queue, three-stage allocation, and fixed-point multi-attention cache-hit search (kv_cache_manager.py + block_pool.py + kv_cache_coordinator.py).
- The persistent InputBatch and decoupled forward/sampling via ExecuteModelState: zero-realloc continuous batching on the GPU plus two-phase execute_model/sample_tokens enabling async scheduling and PP overlap (gpu_model_runner.py + gpu_input_batch.py).
- The ZMQ+msgpack IPC boundary with zero-copy tensor transport: EngineCoreClient/EngineCoreProc, byte-tagged request protocol, and TensorIpcSender/Receiver (core_client.py + core.py + tensor_ipc.py).

### entrypoints

_How inference requests enter vLLM through the offline LLM API and OpenAI-compatible async server, routing them through async wrappers and input processors to the engine core for execution._

**关键文件:**
- `vllm/entrypoints/llm.py` (1937 LOC) — Synchronous offline API exposing generate(), chat(), embed(), and encode() methods that construct EngineArgs, instantiate LLMEngine, and batch requests through internal helpers (_run_completion, _run_chat) before engine execution.
- `vllm/entrypoints/openai/api_server.py` (718 LOC) — OpenAI-compatible FastAPI server entry point that orchestrates lifespan, creates AsyncLLM engine via build_async_engine_client context manager, wires request handlers, and coordinates error handling across all OpenAI API routes.

- `vllm/v1/engine/input_processor.py` (486 LOC) — Converts user-provided PromptType, SamplingParams, or PoolingParams into EngineCoreRequest objects, handling tokenization, multi-modal preprocessing, task routing (generation vs. pooling), and parameter validation.

### async-engine

_AsyncLLM is a 3-stage async facade that decouples input preprocessing, EngineCore scheduling/execution, and output postprocessing into concurrent independent loops, enabling non-blocking streaming inference via per-request async generators and event-driven ZMQ IPC._

**关键文件:**
- `vllm/v1/engine/async_llm.py` (1069 LOC) — Main async facade: wraps EngineCore client, orchestrates 3-stage decoupling via InputProcessor, OutputProcessor, and background output_handler loop
- `vllm/v1/engine/core_client.py` (1695 LOC) — EngineCoreClient hierarchy: AsyncMPClient drives async input/output over ZMQ sockets; manages handshakes, output queue tasks, and async request/abort/utility calls
- `vllm/v1/engine/coordinator.py` (465 LOC) — Data-parallel coordinator: mediates between multiple DP engine ranks and frontend clients, manages wave synchronization and request load stats
- `vllm/v1/engine/core.py` (2145 LOC) — EngineCore and EngineCoreProc: inner scheduler/executor loop; runs in separate process, receives requests via input queue, produces outputs via output queue
- `vllm/v1/engine/input_processor.py` (486 LOC) — Input stage: converts raw prompts (PromptType/EngineInput) and params into EngineCoreRequests; tokenizes, validates, handles streaming inputs
- `vllm/v1/engine/output_processor.py` (812 LOC) — Output stage: converts EngineCoreOutputs to RequestOutput; maintains per-request state, detokenizes tokens, accumulates streaming deltas, pushes to per-request queues

**关键类:**
- `AsyncLLM` (vllm/v1/engine/async_llm.py) — Main async facade: manages lifecycle (init, shutdown), orchestrates add_request and generate flows, spawns background output_handler asyncio task, coordinates InputProcessor and OutputProcessor
- `AsyncMPClient` (vllm/v1/engine/core_client.py) — Async ZMQ client: sends EngineCoreRequests over input socket (add_request_async), receives EngineCoreOutputs via background task consuming output socket, manages output queue for non-blocking async iteration
- `EngineCoreProc` (vllm/v1/engine/core.py) — Background process: runs busy loop consuming EngineCoreRequests from input queue, calls scheduler.schedule(), executor.execute_model(), produces EngineCoreOutputs to output queue via worker threads
- `InputProcessor` (vllm/v1/engine/input_processor.py) — Input preprocessing: validates params, tokenizes prompts, handles multi-modal features, produces EngineCoreRequest objects
- `OutputProcessor` (vllm/v1/engine/output_processor.py) — Output postprocessing and per-request queuing: consumes EngineCoreOutputs, maintains RequestState per request, detokenizes/accumulates tokens, pushes RequestOutput to per-request RequestOutputCollector queues
- `RequestOutputCollector` (vllm/v1/engine/output_processor.py) — Per-request async queue: holds RequestOutput/PoolingRequestOutput for one request, provides async get() and non-blocking get_nowait(), signals ready event when output arrives
- `DPCoordinator` (vllm/v1/engine/coordinator.py) — Data-parallel orchestration: runs in separate process, collects stats from DP engine ranks, broadcasts wave coordination messages, tracks global request load

**v1 设计决策:**
